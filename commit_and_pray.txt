So now we move to the second step which is about commiting our changes or taking a snapshot of them. 

imagine you are working on a directory, you have changed some files, you have created some new files or you have deleted some, 
and now this is the end of the day and you want to kinda save your work or tell git that you are on this checkpoint(like games)
in order to do that we follow this steps: 

1.Always use "git status", this command gives you a summary of what you have done on the current working directory. 
for example this is my status when I am writing this txt file for you:

3.at this point we can save our progress by commiting the changes we have staged by a simple command 
	
	-> git commit -m "Your Message"
we always have to set a message on every commit we do, commit messages must be meaningful and also straight forward. 
if you have worked on a program and added a new feature to it and also debugged another part of the program,
I prefer to commit them separately first add feature to staging area then commit with a good detailed message,
Then add debugged files to staging area and commit them.


It is the whole workflow you'll learn this in 1 day but always consider: 

1.work on how to write good commit messages, if you are working on a team, check their messages, follow their pattern
2.add and commit files which are related to each other, 
add files about "Checkout Logic" -> commit 
add files about "Cart Logic" -> commit

BE NEAT, it helps your team and also after a while when you look back at your project, you won't get dizzy.
